# Step 5 Plan: Autolinking ↔ Build Integration

> Generated by Plan agent. This is the implementation plan for integrating
> expo-modules-autolinking with the build system.

## Approach: Hybrid CLI Command

**Decision:** A new CLI command `npx expo-modules-autolinking autolink-windows`
(like RNW's `autolink-windows`) that runs once before build and handles:
1. `.sln` manipulation (text-based, can't be done from MSBuild)
2. `.vcxproj` patching (ProjectReference injection)
3. `ExpoModulesProvider.g.cs` generation
4. Deploy `.targets` generation

This follows RNW's pattern and avoids trying to manipulate solution/project files from MSBuild targets.

## Key Design Decisions

### ExpoModulesProvider.g.cs Location

Generate a dedicated **`ExpoModulesAutolinked`** project:
```
<appWindowsDir>/ExpoModulesAutolinked/
├── ExpoModulesAutolinked.csproj   (generated — references Core + all module csproj)
└── ExpoModulesProvider.g.cs       (generated — typeof() list)
```

This project:
- References `Expo.Modules.Core.csproj`
- References all discovered module `.csproj` files
- Compiles `ExpoModulesProvider.g.cs` → produces `ExpoModulesAutolinked.dll`
- The C++ host's `FindProviderAssemblyPath` discovers it in `managed/`

### .sln Manipulation

Reimplement a simplified version of RNW's `addProjectToSolution` from
`@react-native-windows/cli/lib-commonjs/utils/vstools.js`. Line-based text
manipulation (`.sln` is not XML). Handles:
- `Project(...) / EndProject` blocks
- `ProjectConfigurationPlatforms` entries
- `NestedProjects` under a "Managed" solution folder

Project type GUIDs:
- C# SDK-style: `{9A19103F-16F7-4668-BE54-9A1E7A4F7556}`
- Solution Folder: `{2150E333-8FDC-42A3-9474-1A3956D46DE8}`

### Deploy .targets Generation

Generate `ExpoModulesAutolinked.g.targets` replacing hand-written `ExpoExampleDeploy.targets`:
- Post-build `Copy` target for all managed DLLs → `$(OutDir)\managed\`
- `<Content DeploymentContent=true>` declarations for MSIX packaging
- PDB files for Debug builds
- `nethost.dll` declaration

## Command Flow

```
npx expo-modules-autolinking autolink-windows \
  --sln <path-to-.sln> \
  --app-proj <path-to-app.vcxproj>

1. Parse args → detect sln, app vcxproj
2. findModulesAsync + resolveModulesAsync (existing pipeline)
3. Generate ExpoModulesAutolinked.csproj
4. Generate ExpoModulesProvider.g.cs
5. Generate ExpoModulesAutolinked.g.targets
6. Update .sln:
   a. Ensure "Managed" solution folder exists
   b. Add Expo.Modules.Core.csproj
   c. Add ExpoModulesAutolinked.csproj
   d. Add each module's .csproj
   e. Nest all under "Managed" folder
7. Update app .vcxproj:
   a. Add ProjectReference to ExpoModulesAutolinked.csproj (with metadata)
   b. Ensure CppWinRTGenerateWindowsMetadata=false
   c. Add Import for ExpoModulesAutolinked.g.targets
8. Remove stale entries (modules that were removed)
```

## Integration with `yarn react-native run-windows`

```
# In example/package.json scripts:
"autolink-expo": "npx expo-modules-autolinking autolink-windows --sln windows/ExpoModulesWindowsCoreExample.sln --app-proj windows/ExpoModulesWindowsCoreExample/ExpoModulesWindowsCoreExample.vcxproj"

# Run before build:
yarn autolink-expo && yarn react-native run-windows
```

## Example Module Changes

Add `expo-module.config.json` to `example/modules/ExampleModule/`:
```json
{
  "platforms": ["windows"],
  "windows": {
    "modules": ["ExampleModules.ExampleModule"],
    "projectPath": "ExampleModule.csproj"
  }
}
```

Remove from app vcxproj:
- Manual `<ProjectReference>` to ExampleModule.csproj
- `<Import>` of ExpoExampleDeploy.targets

## .vcxproj ProjectReference Template

```xml
<ProjectReference Include="ExpoModulesAutolinked\ExpoModulesAutolinked.csproj">
  <ReferenceOutputAssembly>false</ReferenceOutputAssembly>
  <SkipGetTargetFrameworkProperties>true</SkipGetTargetFrameworkProperties>
  <Private>false</Private>
</ProjectReference>
```

## Edge Cases

1. **First build (no generated files):** Run `autolink-expo` first. Clear error message if missing.
2. **Adding/removing modules:** Re-run `autolink-expo`. Regenerates everything.
3. **No Windows modules:** Empty provider, ExpoModulesAutolinked still references Core only.
4. **Platform-specific output paths:** Uses `$(Platform)` in deploy .targets (x64, ARM64).
5. **CppWinRT cycle:** Auto-set `<CppWinRTGenerateWindowsMetadata>false</CppWinRTGenerateWindowsMetadata>`.
