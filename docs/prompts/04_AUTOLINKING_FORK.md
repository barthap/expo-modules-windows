## Context

We're building Expo Modules for Windows/C#. The autolinking system needs to:
1. Recognize "windows" as a supported platform in expo-module.config.json
2. Scan packages for Windows-specific module declarations
3. Generate ExpoModulesProvider.g.cs listing all module classes
4. Integrate with MSBuild (via .targets file)

See docs/RESEARCH_AUTOLINKING.md for full analysis of the existing architecture.

D:\dev\expo/packages/expo-modules-autolinking/

## What the existing platforms do

### iOS (Apple)
- Scans for .podspec files
- Generates ExpoModulesProvider.swift with module class list
- Triggered by: `expo-modules-autolinking generate-modules-provider --platform apple`
- Called from: Ruby script during `pod install`

### Android
- Scans for *Package.java/kt files
- Returns JSON module list (no code generation)
- Triggered by: `expo-modules-autolinking resolve --platform android --json`
- Called from: Gradle plugin

## What to implement for Windows

### 1. Add "windows" to SupportedPlatform type
File: `src/types.ts`
- Add `'windows'` to the SupportedPlatform union type

### 2. Add RawWindowsModuleConfig type
File: `src/types.ts`
```typescript
export type RawWindowsModuleConfig = {
  /** C# module class names (fully qualified) */
  modules?: (string | { name: string; class: string })[];
  /** Path to Windows native project (default: "windows") */
  projectPath?: string;
  /** Only include in debug builds */
  debugOnly?: boolean;
};
```
Add `windows?: RawWindowsModuleConfig` to `RawExpoModuleConfig`.

### 3. Create Windows platform resolver
File: `src/platforms/windows/windows.ts`

Responsibilities:
- `resolveModuleAsync(packageName, revision)`:
  - Read `windows` config from expo-module.config.json
  - If `modules` field exists, use it directly
  - Otherwise, scan windows/ folder for *Module.cs files matching `class XModule : Module`
  - Return ModuleDescriptor with module class names and project path

- `resolveExtraBuildDependenciesAsync()`:
  - Return null (NuGet deps are in .csproj directly)

- `generateModulesProviderAsync(modules, targetPath)`:
  - Generate ExpoModulesProvider.g.cs from module list (see template below)

### 4. Generated file: ExpoModulesProvider.g.cs

Template:
```csharp
// Automatically generated by expo-modules-autolinking.
// Do not edit this file manually.

using System;
using System.Collections.Generic;

namespace ExpoModulesWindows
{
    public static class ExpoModulesProvider
    {
        public static IReadOnlyList<Type> GetModuleClasses()
        {
            return new Type[]
            {
${moduleEntries}
            };
        }
    }
}
```

Where `moduleEntries` is lines like:
```
                typeof(ExpoCamera.CameraModule),
                typeof(ExpoImage.ImageModule),
```

Handle debug-only modules with `#if DEBUG` preprocessor:
```csharp
#if DEBUG
                typeof(ExpoDevMenu.DevMenuModule),
#endif
```

### 5. Register in platform router
File: `src/platforms/index.ts`
Add `case 'windows': return require('../platforms/windows');`

### 6. Add ExpoModuleConfig support
File: `src/ExpoModuleConfig.ts`
Add `windowsConfig()` method to return parsed Windows config.

### 7. CLI integration
The `generate-modules-provider` command already supports `--platform`.
Ensure it works with `--platform windows`.

Usage:
```bash
npx expo-modules-autolinking generate-modules-provider \
  --platform windows \
  --target windows/ExpoModulesProvider.g.cs
```

### 8. MSBuild integration hook
Create a template .targets file that consuming apps include:
```xml
<Target Name="GenerateExpoModulesProvider" BeforeTargets="CoreCompile"
        Inputs="$(ProjectDir)..\..\package.json"
        Outputs="$(ProjectDir)ExpoModulesProvider.g.cs">
  <Exec Command="npx expo-modules-autolinking generate-modules-provider --platform windows --target &quot;$(ProjectDir)ExpoModulesProvider.g.cs&quot;" />
</Target>
```
