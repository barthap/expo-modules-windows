## Key constraints

- **CppWinRT cycle**: C# SDK-style projects in a C++/WinRT solution can cause
  cyclic dependency on `CppWinRTComputeGetResolvedWinMD`. Fixed by setting
  `<CppWinRTGenerateWindowsMetadata>false</CppWinRTGenerateWindowsMetadata>`
  on the app vcxproj (when it has no .idl files). The autolinking tool should
  handle this automatically.
- **ProjectReference metadata**: C++ vcxproj referencing C# csproj needs special
  metadata (ReferenceOutputAssembly=false, etc.) to prevent MSBuild from trying
  to resolve .NET framework properties.
- **C# output paths include $(Platform)**: When built from solution context, C#
  projects output to bin\x64\Debug\... (not bin\Debug\...). The deploy targets
  must account for this.
- **MSIX Content declarations**: Files must be `<Content>` with
  `<DeploymentContent>true</DeploymentContent>` to appear in the AppX package.
  Post-build Copy targets only reach $(OutDir), NOT the deployed package.

## Reference: How RNW autolink-windows updates .sln

The `@react-native-community/cli autolink-windows` command:
1. Reads react-native.config.js from each package
2. Finds .vcxproj files for each native module
3. Adds project entries to the .sln
4. Adds configuration mappings for all platform/config combos
5. Generates AutolinkedNativeModules.g.cpp with #include and registration code
6. Generates AutolinkedNativeModules.g.props with project references
7. Generates AutolinkedNativeModules.g.targets with build dependencies

We should follow a similar pattern but for C# projects.
