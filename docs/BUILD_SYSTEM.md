# Build System & Project Structure

This document describes the build shape that exists in the repo today.

## Solution & Project Map

```
ExpoModulesWindowsCoreExample.sln  (example/windows/ — main app solution)
├── ExpoModulesWindowsCoreExample.vcxproj          C++ Win32 RNW app
├── ExpoModulesWindowsCoreExample.Package.wapproj  MSIX packaging project
├── ExpoModulesWindowsCore.vcxproj                 Native library project
└── Managed/                                       Solution folder
    ├── Expo.Modules.Core.csproj
    ├── ExpoModulesAutolinked.csproj               Generated by Expo autolinking
    └── ExampleModule.csproj

ExpoModulesWindowsCore.sln  (windows/ — lib-only work)
└── ExpoModulesWindowsCore.vcxproj
```

The example solution now includes managed projects under a `Managed` solution
folder. The app itself only takes a direct project reference to the generated
`ExpoModulesAutolinked.csproj`, which in turn references `Expo.Modules.Core`
and all discovered module projects.

## What Each Project Produces

| Project | Type | Output |
|---------|------|--------|
| `ExpoModulesWindowsCore.vcxproj` | C++ DLL | `ExpoModulesWindowsCore.dll` + `.winmd` (includes vendored `common/cpp/` from expo-desktop) |
| `ExpoModulesWindowsCoreExample.vcxproj` | C++ EXE | `ExpoModulesWindowsCoreExample.exe` |
| `ExpoModulesWindowsCoreExample.Package.wapproj` | MSIX Package | `AppX/` layout directory |
| `Expo.Modules.Core.csproj` | C# Class Library | `Expo.Modules.Core.dll` + `.runtimeconfig.json` |
| `ExpoModulesAutolinked.csproj` | Generated C# hub project | `ExpoModulesAutolinked.dll` |
| `ExampleModule.csproj` | C# Class Library | `ExampleModules.dll` |

## Two Autolinking Pipelines

The Windows app depends on two separate codegen/autolinking steps:

1. **React Native Windows autolinking**
   `npx @react-native-community/cli autolink-windows`
   This generates `AutolinkedNativeModules.g.{h,cpp,props,targets}` so RNW can
   register `ExpoModulesWindowsCore::ReactPackageProvider`.

2. **Expo Windows autolinking**
   `npx expo-modules-autolinking autolink-windows --sln ... --app-proj ...`
   This generates:
   - `ExpoModulesAutolinked/ExpoModulesAutolinked.csproj`
   - `ExpoModulesAutolinked/ExpoModulesProvider.g.cs`
   - `ExpoModulesAutolinked.g.targets`
   and patches the example `.sln` and app `.vcxproj`.

These are complementary:
- RNW autolinking registers the native package with React Native Windows.
- Expo autolinking discovers C# Expo modules and wires the managed build graph.

## Current Workflow Caveat

The repo is currently in a transitional but working state:

- The example project files already reference the Expo-generated outputs.
- The generated Expo outputs are intentionally gitignored.
- `run-windows` does not yet invoke the Expo autolinking command automatically.

That means a fresh clone should run the Expo autolinking command before the
first Windows build and after adding or removing Windows Expo modules.

## `run-windows` vs F5 in Visual Studio

`yarn react-native run-windows` (via `@react-native-community/cli`) does:

1. Run RNW autolinking (`@react-native-community/cli autolink-windows`)
2. Build the solution
3. Deploy the MSIX package
4. Launch the packaged app

It does **not** currently run `expo-modules-autolinking autolink-windows` for
you, so that remains a manual pre-step when managed module references change.

**F5 in VS** skips both autolink commands unless the generated files are already
present and up to date. For Visual Studio debugging:

1. Run both autolink commands when needed
2. Set `ExpoModulesWindowsCoreExample.Package` as the startup project
3. Build and launch via the `.Package` project, not the raw `.vcxproj`

## MSBuild Targets: How Managed Code Reaches the App

### `ExpoManagedDeploy.targets` (library project)

Imported by `windows/ExpoModulesWindowsCore/ExpoModulesWindowsCore.vcxproj`.

Responsibilities:
- build-order-safe copying of the core managed outputs to `$(OutDir)\managed\`
- package `nethost.dll` with the native library output

This target supports the standalone library project.

### `ExpoModulesAutolinked.g.targets` (app project)

Imported by the example app vcxproj after Expo autolinking.

Responsibilities:
- copy all managed outputs to `$(OutDir)\managed\`
- declare them as `Content` with `DeploymentContent=true` for MSIX packaging
- include PDBs in Debug builds
- ensure `nethost.dll` is packaged

This replaces the older hand-maintained example-specific deploy target approach.

## Why `DeploymentContent=true` Matters

The `.wapproj` deploys from the AppX layout, not directly from `$(OutDir)`.
Files copied only by a post-build step are not automatically present in the
deployed package.

Any runtime file that the packaged app needs must be declared as `Content` with
`DeploymentContent=true`, including:

- `nethost.dll`
- managed assemblies in `managed\`
- `Expo.Modules.Core.runtimeconfig.json`

Without those declarations, the packaged app may fail before your own module
code runs because the native module DLL cannot load its dependencies.

## Native Package Registration

RNW still uses its normal generated package registration flow:

1. `AutolinkedNativeModules.g.cpp` registers `ExpoModulesWindowsCore::ReactPackageProvider`
2. RNW creates the single native package
3. The package initializes the Expo JSI HostObject
4. The C++ host loads the managed runtime and discovers module types from
   `ExpoModulesAutolinked.dll`

## CppWinRT Cycle Notes

The original build work hit a circular dependency problem when mixing C# and
C++/WinRT projects. The current working shape avoids that by:

- using a single generated app-side managed hub project
- using `ProjectReference` metadata tuned for C++ project consumption
- forcing `CppWinRTGenerateWindowsMetadata=false` on the app vcxproj

The standalone library solution intentionally remains minimal and does not
mirror the example app's managed project layout.

## Vendored C++ Layer

The `windows/ExpoModulesWindowsCore/common/cpp/` directory contains Expo's shared C++ runtime layer, vendored from [expo-desktop](https://github.com/shirakaba/expo-desktop) (Expo SDK 54, MSVC-patched). These files compile with `<PrecompiledHeader>NotUsing</PrecompiledHeader>` since they are cross-platform code that doesn't use Windows PCH.

The vcxproj adds `$(ReactNativeWindowsDir)..\react-native\ReactCommon` to include paths for the `cxxreact/ErrorUtils.h` dependency used by `EventEmitter.cpp`.

See [EXPO_DESKTOP.md](EXPO_DESKTOP.md) for details on versioning, updating, and our relationship with expo-desktop.
