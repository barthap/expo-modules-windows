## 1. Autolinking Architecture Overview

### How Autolinking Discovers Packages

**Discovery Process:**
1. **Module Search**: Scans `node_modules` recursively and custom search paths defined in `package.json` via `expo.autolinking` config
2. **Config Detection**: Looks for `expo-module.config.json` or `unimodule.json` in each package root
3. **Platform Filtering**: Only includes packages that explicitly support the target platform via `"platforms"` array
4. **Dependency Linking**: By default includes transitive dependencies (SDK 54+); can be restricted with `legacy_shallowReactNativeLinking` flag

**Entry Points for Discovery:**
- `packages/expo-modules-autolinking/src/autolinking/findModules.ts` — Main discovery logic
- `packages/expo-modules-autolinking/src/ExpoModuleConfig.ts` — Config parsing & platform support detection

## 2. expo-module.config.json Schema

### Complete Structure

```typescript
// Location: src/types.ts lines 281-367

export interface RawExpoModuleConfig {
  /**
   * Array of supported platforms
   * Current: "apple" | "ios" | "macos" | "tvos" | "android" | "web" | "devtools" | (string & {})
   * Can be extended to include "windows"
   */
  platforms?: SupportedPlatform[];

  apple?: RawModuleConfigApple;    // iOS, macOS, tvOS
  ios?: RawModuleConfigApple;      // Legacy (fallback for 'apple')
  android?: RawAndroidConfig;
  coreFeatures?: string[];         // e.g., "swiftui", "jetpackcompose"
  devtools?: { ... };
}
```

### Apple Platform Config

```typescript
export type RawModuleConfigApple = {
  modules?: (string | { name: string; class: string })[];  // Swift module classes
  appDelegateSubscribers?: string[];   // ExpoAppDelegate lifecycle hooks
  reactDelegateHandlers?: string[];    // ExpoReactDelegateHandler classes
  podspecPath?: string | string[];     // Podspec relative path(s)
  swiftModuleName?: string | string[]; // Swift product module names for imports
  debugOnly?: boolean;
};
```

### Android Platform Config

```typescript
export type RawAndroidConfig = {
  name?: string;
  path?: string;
  modules?: (string | { name: string; class: string })[];  // Fully qualified class names
  services?: string[];                  // Service classes extending expo.modules.kotlin.services.Service
  publication?: AndroidPublication;
  gradleAarProjects?: AndroidGradleAarProjectDescriptor[];
  projects?: WithRequired<RawAndroidProjectConfig, 'name' | 'path'>[];  // Multiple subprojects
  gradlePlugins?: AndroidGradlePluginDescriptor[];
};
```

## 3. Platform Resolver Contract

Each platform implements (inferred from implementations):

```typescript
export interface PlatformResolver {
  resolveModuleAsync(
    packageName: string,
    revision: PackageRevision,
    extraOutput?: { flags?: Record<string, any> }
  ): Promise<ModuleDescriptor | null>;

  resolveExtraBuildDependenciesAsync(
    projectNativeRoot: string
  ): Promise<ExtraDependencies | null>;

  // Only Apple:
  generateModulesProviderAsync?(
    modules: ModuleDescriptor[],
    targetPath: string,
    entitlementPath: string | null
  ): Promise<void>;

  // Only Android:
  getConfiguration?(options: AutolinkingOptions): Record<string, any> | undefined;
}
```

## 4. Code Generation

### iOS/Apple: Generated `ExpoModulesProvider.swift`

**Triggered by:** `expo-modules-autolinking generate-modules-provider --target <path> --platform apple`
**Called from:** Ruby script during `pod install`

```swift
// Automatically generated by expo-modules-autolinking.

internal import ExpoModulesCore
internal import EXCamera
internal import ExpoImage

@objc(ExpoModulesProvider)
internal class ExpoModulesProvider: ModulesProvider {
  public override func getModuleClasses() -> [ExpoModuleTupleType] {
    return [
      (module: CameraViewModule.self, name: nil),
      (module: ImageModule.self, name: nil)
    ]
  }

  public override func getAppDelegateSubscribers() -> [ExpoAppDelegateSubscriber.Type] {
    return [
      CameraAppDelegate.self
    ]
  }

  public override func getReactDelegateHandlers() -> [ExpoReactDelegateHandlerTupleType] {
    return [
      (packageName: "expo-camera", handler: CameraReactDelegate.self)
    ]
  }
}
```

**Key features:**
- Uses Swift preprocessor: `#if EXPO_CONFIGURATION_DEBUG` for debug-only modules
- Module name override: `(module: Class.self, name: "customName")` when name differs from default

### Android: No Code Generation

Android uses a **data-driven approach**:
- `resolve --platform android --json` returns JSON module list
- Gradle plugin processes the JSON at build time
- Modules discovered at runtime, not via generated source

## 5. Build System Integration

### iOS Flow
1. User runs `pod install`
2. CocoaPods invokes Ruby script hook (in expo-modules-core)
3. Hook calls: `expo-modules-autolinking generate-modules-provider --target ExpoModulesProvider.swift`
4. Generated Swift file included in build

### Android Flow
1. Gradle build starts
2. Gradle plugin (from expo-modules-core) calls: `expo-modules-autolinking resolve --platform android --json`
3. Returns JSON module list
4. Plugin processes at build time

## 8. Proposed Windows Platform Design

### expo-module.config.json — Windows Section


### Build Integration Flow

1. App builds: `dotnet build` or MSBuild via Visual Studio
2. MSBuild `.targets` runs prebuild task
3. Task calls: `expo-modules-autolinking resolve --platform windows` (or `generate-modules-provider`)
4. Autolinking scans packages, generates `ExpoModulesProvider.g.cs`
5. C# compiler includes generated file
6. C++ host loads the .NET assembly and reads the module list

### Comparison Table

| Aspect | iOS/Apple | Android | Windows (proposed) |
|--------|-----------|---------|-------------------|
| **Discovery trigger** | `pod install` (Ruby) | Gradle plugin | MSBuild target (prebuild) |
| **Code generation** | Yes — Swift file | No — JSON data | Yes — C# file |
| **Generated file** | `ExpoModulesProvider.swift` | N/A | `ExpoModulesProvider.g.cs` |
| **Build integration** | CocoaPods | Gradle plugin | MSBuild .targets |
| **Module config key** | `apple` | `android` | `windows` |
| **Module format** | Swift class name | Fully qualified Kotlin class | Fully qualified C# class |

### Windows Resolver Implementation

New file: `src/platforms/windows/windows.ts`

Key responsibilities:
1. Look for `windows/` folder (or custom path from config) containing `.csproj` files
2. Scan for `*Module.cs` files matching `class XyzModule : Module` pattern
3. Return module descriptors with project paths and class names
4. Generate `ExpoModulesProvider.g.cs` (similar to Apple's Swift generation)
